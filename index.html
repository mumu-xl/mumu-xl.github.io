对RAG的探索：从知识库治理到高效召回

Agent是什么？Agent = RAG + LLM + Tools

对于我们大模型来说，从它被训练出来，他的知识就已经固化了，所以对于它没有的知识或者有些我们自己整理的数据就需要搭建一个知识库来通过RAG进行检索，召回丢给大模型让他整理回答，所以RAG的重要性不言而喻。

所以 RAG 真正难的地方，不是“怎么接一个向量库”，而是：如何让知识以更容易被检索、被判断、被更新的方式进入系统。

这篇文章结合 RAG 高级技术与调优的内容，整理一下我现在对 RAG 的理解。

RAG 的三个基本步骤

RAG 可以拆成三个阶段：

  1. Indexing：如何把知识存起来
  2. Retrieval：如何从大量知识中找到一小部分有用内容
  3. Generation：如何结合用户问题和检索结果生成答案

第一层：先把知识库处理好

1. 为知识切片生成问题

首先对于我们的数据，格式各种各样，如 text，pdf，office等等，我这边采用的操作读取所有的数据，然后转化成markdown格式，因为大模型天然对文本类型支持性更强。

接下来就是对我们的markdown数据进行重组变成chunk，chunk的结构是这样的。

想要把数据切的更加的完整，这关乎到我们接下来的召回是否准确。

  • 通过文档结构进去切割
  • 将我们的文档切割成一小块一小块，然后循环把每一个chunk都embendding一下，获取他的向量值，相邻的向量值我们去计算它的余弦值，值大于0.9 我们觉得它俩高度相似。就可以进行把这俩进行一个合并，整个流程跑下来，就把所有相似的内容切切割成一个chunk，这种方法稍微消耗tokens
  • 使用GraphRAG，这个项目是谷歌开源的工具，主要是对我们的数据做了汇总和总结，具体没有多了解，它的优势就是召回的数据准确度非常高，当然这个非常贵，往往一个问题话费几毛钱的tokens，如果是比较值钱的数据，可以采用这个方法

我使用的是第一种方式，通过文档结构进行切割，在metadata中把我们的文件名称，标题等拼接进去，

切割成上面这种格式就差不多了，接着就是通过embedding模型将数据转成向量存进向量数据库中

2. 数据召回

数据召回阶段也很重要，我采用了embedding模糊匹配和BM25关键词查询两种方式，每一种方式召回十条数据
为什么要这么做呢？
对于embedding来说，他是一种相似度计算来进行匹配查询，因为之前把数据转成向量值存进向量数据库中，它的操作就是通过我们的问题去库中计算两者的余弦值，余弦值越大说明越相似，但是这里也有一个问题
比如 问：苹果好吃吗？
模型可能返回：香蕉很好吃
这俩相似度很高，但是和我们想要的结果天差地别
所以就需要BM25来继续查询一次，这个是将我们的问题拆分成关键词，通过关键词去查询metadata中的关键词，
每一种我这边设置的是返回10条给到我们

3. Rerank打分

Rerank模型可以将我们的问题和chunk进行打分，把和问题最接近的答案分数会打的更高，我们一般是把召回的数据和答案给到大模型让他打分，而不是把所有数据都给它，这样会比较省tokens。

我们拿到最终结果以后，取分数最高的前几个，拼接成提示词，给到大模型，让他根据提示词去生生最终答案，这样可以提高我们的准确性

然后我们去向相同的大模型问同样的问题

OpenSpec+deepseek 漫剧内容站实战总结

用 AI 写代码的人，应该都有过这种体验：一句话需求丢过去，AI 一顿猛写，方向跑偏，你得来回拉扯好几轮才能把代码掰回来。

更要命的是，同样一句话，给 Claude 是一个结果，给 DeepSeek 又是另一个结果，而且质量差距不小。久而久之就变成了”只用最强模型”——成本高，速度慢。

我现在的解法是：把 OpenSpec 作为核心锚点，配合 DeepSeek + Claude Code + CC Switch 打配合。 这篇文章主要讲 OpenSpec——它是什么、怎么用、生成什么文件、有哪些命令，以及为什么加上它之后，连国产大模型也能写出很标准的代码。

先简单认知一下其他三个工具

DeepSeek：国产大模型，推理风格偏严谨保守，擅长逻辑分析和方案盘点。成本低。

Claude Code：工程能力最强的 AI 编程工具，擅长写复杂逻辑、重构、调试。

CC Switch：一键切换 Claude Code 使用的模型和配置。不同任务用不同模型（画页面换Kimi，写需求用deepseek，或者最新的小米大模型）。

这三个工具单独用都挺好，但有一个问题没解决：需求本身是乱的，模型再强也白搭。

这就是 OpenSpec 出场的地方。

OpenSpec 是什么

OpenSpec 是一个**规范驱动开发（Spec-Driven Development）**的 CLI 工具和框架

它的核心思路很简单：在写任何代码之前，先让人和 AI 就”要构建什么”达成一致。 用结构化的 Markdown 规范文档作为单一真相源，AI 照着规范实现，而不是凭空猜。

安装完之后，在你的项目根目录执行：

初始化时会让你选择对接的 AI 工具（支持 Claude Code、Cursor、Windsurf、Gemini CLI、GitHub Copilot 等 21+ 种），然后它会帮你建好一整套目录结构。

OpenSpec 生成了什么：目录与文件结构

初始化之后，你的项目里会多出一个 openspec/ 目录，结构如下：

每个文件是干什么的：

  • proposal.md：变更提案。说清楚为什么要做、做什么、影响范围、风险评估。这是在动手前让所有相关方（包括 AI）对齐认知的文件。
  • tasks.md：实施任务清单。把变更拆成一个个可执行的小步骤，比如”创建 User 模型”、”添加 POST /login 接口”、”编写登录测试”。AI 按这个清单一条一条执行。
  • design.md：技术设计（可选）。当变更涉及架构调整、新技术引入时，把设计决策和技术选型写在这里。
  • spec.md（增量）：本次变更要新增或修改的规范内容。一个 capability 对应一个 spec.md。这里不会重复写已有的规范，只写 delta——本次变更相对现状的差异。

归档之后，changes/ 里的增量 spec 会合并到 openspec/specs/ 下的主规范中，成为系统的新”真相源”。

OpenSpec 有哪些命令

OpenSpec 提供了两套命令：终端 CLI 命令 和 AI 聊天界面里的斜杠命令。

终端 CLI 命令

命令 作用
openspec init 初始化项目结构和 AI 工具集成
openspec list 列出所有进行中的变更
openspec list --specs 列出所有已归档的规范能力
openspec show <change-id> 查看某个变更的详细内容
openspec view 启动交互式 TUI 可视化仪表盘
openspec validate <change-id> 校验规范文档格式是否正确
openspec validate <change-id> --strict 严格模式校验
openspec archive <change-id> 将完成的变更合并到主规范库并归档
openspec version 查看版本

日常最常用的是 list、show、validate、archive 这四个。validate 尤其重要——它像 lint 一样检查你的 spec.md 格式对不对，避免格式错误导致 AI 理解偏差。

AI 斜杠命令（在 Claude Code 等工具里直接使用）

命令 作用
/opsx:explore <idea> 探索阶段：在创建文件之前，先梳理想法、检查现有代码库
/opsx:propose <feature> 提案阶段：生成 proposal.md、design.md、tasks.md 和规范增量 spec.md
/opsx:apply 实施阶段：AI 按 tasks.md 中的清单逐步编码实现
/opsx:archive 归档阶段：将完成的变更合并到 specs/ 并移至 archive/

为什么 OpenSpec 能让国产大模型写出标准化代码

这是这篇文章最想说的点。

国产大模型（DeepSeek、通义千问、文心一言等）和 Claude / GPT 相比，最大的差距在哪？

我自己的体验是：不是”写代码”这个动作本身差多少，而是”理解你想要什么”的能力有差距。

你给一个模糊的需求，Claude 能猜出大致方向，质量还算过得去；但国产模型可能会偏得比较厉害。这不是说它们代码能力弱，而是它们在高层次意图理解上做了不同的权衡。

OpenSpec 恰好解决了这个问题——它把”理解你想要什么”这一步，从 AI 那里剥离出来，前置到 spec 文档里了。

具体来说，它从四个层面让任何模型都能输出标准化代码：

1. 输入从模糊变精确。 以前 AI 拿到的是”帮我调接口”——这里面可猜的东西太多了。现在它拿到的是 spec.md，里面 API 签名、约束条件、边界情况、返回格式、错误码全写死了。AI 不需要猜，只需要实现。对于逻辑能力强但理解能力偏弱的模型来说，这个差距是决定性的。

2. 任务拆分减少了单次复杂度。 tasks.md 把一个变更拆成了小步骤。以前是”一口气写 500 行出来”，现在是”画这个页面，写好再写下一个”。每一步都有明确的输入输出和约束，出错的概率大幅降低。

3. 规范不挑模型。 spec.md 是纯 Markdown，不依赖任何特定 AI。你可以在 DeepSeek 上跑 /opsx:apply，它照着 spec 实现一套代码；再切到 Claude 上跑一次，实现出来的方向也是一致的。因为规范本身就是唯一的”设计要求”，模型只是执行者。

说白了，OpenSpec 的范式是：

不确定性最大的那一步——“理解你要什么”——被规范文档承接了。剩下的”按规范执行”是几乎所有大模型都能做好的事。

总结

工具 角色 一句话
OpenSpec 规范锚点 把”要做什么”写清楚，让任何模型都能按图施工
DeepSeek 审稿 + 执行 审查规范逻辑、盘点方案，拿到 spec 后也能写出标准代码
Claude Code 主力工程 复杂重构、架构调整，工程能力最强
CC Switch 模型调度 不同任务一键切不同模型，提效降本

这套流程跑下来，最大的感受不是”更快了”，而是代码质量变得稳定了——不会因为换了一个模型、换了一个时间、换了一个功能，码风和质量就忽上忽下。

浏览器执行原理和性能提升

浏览器是如何下载js文件的呢？

首先在浏览器中输入域名，然后这个域名将转化服务器的IP地址，这个IP地址将返回一个文件，就是图中的index.html文件，这个时候浏览器开始解析这个网页，解析到link标签就下载css文件，解析到script标签就下载js文件。

浏览器的渲染过程

浏览器将html，css，js文件下载下来后就开始解析，这个时候由浏览器内核中的HtmlParse模块将HTML标签编译成DOMTree，这个时候可以利用js进行DOM操作将HTML元素插入到DOMTree中，而浏览器则是通过浏览器引擎来解析js的，我们在下面的内容将会重点将浏览器V8引擎。
DOMTtree生成以后将会开始解析css文件，通过cssParse模块将css解析为StyleRulesTree，接下来DOMTree和StyleRulesTree将会结合形成RenderTree。

RenderTree生成以后通过Layout引擎根据浏览器大小来生成新的renderTree，然后绘制出来，浏览器开始展示。由于有些html标签定位不同，这个时候浏览器大小不同，元素位置就不同，layout引擎的作用就是重新生成一个适合当前浏览器大小的renderTree。

V8引擎

JS引擎

高级的编程语言都是需要转成最终的机器指令来执行的

事实上我们编写的JavaScript无论你交给浏览器或者Node执行，最后都是需要被CPU执行的

但是CPU只认识自己的指令集，实际上是机器语言，才能被CPU所执行；

所以我们需要JavaScript引擎帮助我们将JavaScript代码翻译成CPU指令来执行

浏览器V8引擎是如何编译js的？

/
首先将js源代码通过parse模块编译成抽象语法树，之后通过Ignition模块将抽象语法树编译成字节码，然后V8引擎将字节码转化成汇编代码，汇编代码转化为对应平台的机器指令运行。

当一个函数多次执行时，Ignition模块将会把这个函数标记成一个hot函数，由TurboFan编译成优化的机器码，这样当下一次执行这个函数，就直接执行这串机器码。当然有一个前提，这个函数传递的参数类型必须一致。

CodeX + Skill 学习python一周进展

前端学后端，最容易出现的问题：

以前只能根据视频去学习，或者自己看技术文章。而视频只会告诉你这么做，但是不会告诉你为什么这么做，主打一个速成。后续如果不继续接触，就会遗忘。而看别人的技术文章不一定是自己需要的，而且还不一定能看懂。

而通过ai去学习，搭配skil，让它提供思路，从先实现一个功能，再到去设计这个功能，怎么让它变得更合理，更规范更易于维护。
从一个前端开发者的真实视角，聊聊我怎么借着一个 Flask 项目，在 CodeX + Skill 的帮助下，一步一步把后端这件事想清楚。

起步

刚开始写 Flask 的时候，我最自然的反应就是：

先让接口跑起来。

这个思路放在前端其实挺常见的。页面先显示出来，交互先通，细节后面再补。
但到后端，如果这个习惯一直延续，就很容易把所有东西都堆在一个接口里。

比如我一开始写登录，大概就会写成这种风格：

现在回头看，这段代码几乎把我当时的思路暴露得很彻底：

  • 路由里直接查数据库
  • 密码直接和数据库字段比较
  • 返回结构想到什么写什么
  • 业务逻辑和请求处理混在一起
  • 后面如果要接 token、鉴权、测试，基本都得重新梳理

但那个阶段的我，其实是觉得“这很合理”的。
因为我当时还没有真正建立后端项目里的分层意识。

后来慢慢发现，我缺的不是代码，是对结构的判断

这也是 CodeX + Skill 对我帮助最大的地方。

如果只是普通地问 AI：

  • 帮我写个登录接口
  • 帮我修这个报错
  • 帮我加个 token 鉴权

通常也能解决问题。
但这种方式很容易让我停在“拿到结果”这一层。

但是对于初学者的我来说，更重要的是：

  • 这段逻辑为什么不适合放在这里？
  • 这层代码到底在负责什么？
  • 这个值我应不应该信任前端传进来？
  • 这个错误是业务失败，还是数据库本身出了问题？

Skill 最让我舒服的一点，是它不会老把我往“直接堆功能”那条路上带，而是会一直把注意力拉回到：

你现在写的这段代码，到底有没有边界感。

这个东西，前端开发者在转后端的时候，真的特别需要。

我是从“登录注册”这种最基础的功能里，慢慢理解后端的

我这个项目是 Flask + Flask-SQLAlchemy，用户模块是最先做起来的。

一路做下来，大概包括这些东西：

  • 注册
  • 登录
  • 获取用户详情
  • 修改密码
  • token 鉴权
  • 统一鉴权装饰器
  • 接口测试

这些功能看上去都不大，但它们其实把后端里很多关键知识都串起来了。

后面我慢慢把结构整理成了这样：

  • views：接收请求，返回响应
  • service：做业务判断
  • dao：负责数据库操作
  • schema：控制返回结构
  • utils：放 token 和通用能力

最开始我其实对这种拆法没有太多感觉，甚至会觉得“只是多绕了一层”。
但等我真的开始一个功能一个功能往里加的时候，才发现它的价值非常具体。

比如登录功能，第一次真正意识到“分层”不是形式问题

后面把登录逻辑逐步拆开之后，大概是这样的。

路由层

这层做的事情非常明确：

  • 接 JSON
  • 做最外层格式校验
  • 调 service
  • 统一返回响应

到了这里，我第一次开始有点后端的感觉。
因为我终于不再把“拿请求”和“做业务判断”写在一起了。

业务层

这里我才真正开始做“后端判断”：

  • 参数是否为空
  • 用户是否存在
  • 密码是否正确
  • 登录成功后要返回什么

以前做前端的时候，我对“业务层”这个概念其实没那么强烈。
更多时候是状态和交互混着走。
但后端不一样，一旦这里不分开，后面接口一多，代码会很快失控。

数据访问层

这层给我的感觉特别像前端里“只做一件事的基础层”。

它只回答一个问题：

  • 数据库里有没有这个用户

它不应该负责：

  • 用户能不能登录
  • 这个错误要怎么提示
  • token 要不要生成

这种层和层之间的边界感，是我以前在前端项目里不太会强迫自己思考的。
但做后端的时候，这件事会越来越重要。

Skill 的作用

CodeX + Skill 真正帮到我的，是帮我改正代码中的不规范。

它让我不会那么轻易地糊弄过去。

比如我现在会更习惯去问自己：

  • 这段逻辑写在 views 里，是不是已经开始越界了？
  • 这个错误提示看起来没问题，但它掩盖的是数据库冲突还是系统异常？
  • 这个接口已经有 token 了，为什么还要信前端 body 里的用户信息？
  • 这个函数现在能跑，但以后如果复用，它的边界是不是已经乱了？

以前我更容易问的是：

  • 这个接口怎么写
  • 这个 bug 怎么修
  • 这个功能怎么做出来

现在我更关心的是：

  • 这段逻辑为什么应该在这里
  • 这里到底有没有更合理的职责划分

我觉得这是我从前端向后端转的时候，一个特别重要的变化。
